# ADR-0005 — Managed Identity + Key Vault References Over Secrets in Config

**Status:** Accepted
**Date:** 2026-06
**Context:** Lab 02 — Web Platform

## Context

The application needs a SQL database connection string. The original approach was a credential pasted into a config file. A secure approach was needed where no secret lives in code or plaintext application settings, and ideally where the application handles no credential at all.

## Decision

Store the connection string in **Key Vault**, give the App Service a **system-assigned managed identity** with the **Key Vault Secrets User** role, and inject the secret into the app via a **Key Vault reference** app setting (`@Microsoft.KeyVault(SecretUri=...)`).

## Rationale

- **No secret in code or config.** The app setting holds a *pointer* to the secret, not the value. App Service resolves it at runtime using the managed identity.
- **No human handles the credential.** The password is generated by Terraform and stored directly in Key Vault; nobody copies or pastes it.
- **Rotation without redeploy.** Updating the secret in Key Vault is picked up without changing the app.
- **Language-agnostic, zero SDK code.** Key Vault references work for any runtime, unlike fetching secrets via an SDK in app code.

**System-assigned (not user-assigned) identity** was chosen because the identity's lifecycle is tied 1:1 to the app — created and destroyed with it. User-assigned would be preferred only if multiple resources shared one identity.

**Key Vault RBAC mode (not access policies)** was chosen for consistency with the rest of Azure RBAC, inheritance support, and because it is Microsoft's recommended model.

## Consequences

- **Positive:** Eliminates the most common credential-leak vector; no secret in the repo or config; clean rotation story.
- **Trade-off / known limitation:** The Terraform-generated password lands in the Terraform **state file**, which is why the state backend is secured (RBAC-only, encrypted, versioned) and excluded from git. A further hardening step would be Entra-only authentication to SQL (no password at all); the password approach was used here to demonstrate the Key Vault reference pattern.
- **Observed gotcha:** the app's role assignment must exist before the app resolves the reference; a `depends_on` enforces creation order, and RBAC propagation delay can briefly show an unresolved reference.
